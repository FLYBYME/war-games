# War-Games Engine V3: Architecture & Onboarding Guide

## 1. The Target: What Are We Building?
War-Games is a high-fidelity, scientific-grade tactical simulation engine. The target is to build an environment capable of modeling real-world physics, aerodynamics, electronic warfare, and naval combat dynamics in true 3D space (6-Degrees-of-Freedom).

Unlike traditional game engines (which prioritize 60FPS visual approximations), this engine prioritizes **Absolute Determinism, Auditability, and Mathematical Rigor**. If a missile misses a target, an engineer must be able to query the exact physical variables (air density, radar cross-section, jamming power, seeker FOV) at the exact micro-tick that caused the failure.

## 2. Core Philosophy: The Command-Driven ECS
The engine is built on an **Entity-Component-System (ECS)** architecture, but with a critical restriction: **Systems cannot mutate State directly.**

### Why? (The Problem with V2)
In the previous architecture (V2), any system could reach into global memory and change a value at any time. This caused invisible race conditions:
1. The `GuidanceSystem` decides a missile should turn toward a target.
2. A millisecond later in the same tick, the `CombatSystem` decides the target's track was lost and clears the target ID.
3. The missile stops turning for no obvious reason. Depending on the order the systems ran, the simulation behaved differently.

### The Solution: V3
In V3, the simulation tick is split into a **Pure Read Phase** and a **Consolidated Write Phase**.

#### Entities & Components (The Data)
An `Entity` is just a container with a unique ID. A `Component` is a class holding pure data (no logic).
*   **Example Components:** `TransformComponent` (Position/Rotation), `KinematicsComponent` (Velocity/Mass), `AeroComponent` (Wing Area, Lift/Drag Coefficients).

#### Systems (The Logic)
A `System` evaluates the state of the world but **never** changes it. Instead, it returns an array of `Command` objects (Intents).
*   **Example:** Instead of `entity.heading = 90`, the `GuidanceSystem` returns `new SetHeadingCommand(entity.id, 90)`.

#### Commands & The Resolver (The Mutation)
At the end of a phase, the `World` gathers all `Commands` generated by all systems, sorts them by priority, and applies them to the entities.
*   **Conflict Resolution:** If two systems disagree, the Command with the highest priority wins.
*   **Traceability:** The `Tracer` automatically records every applied Command. If a test fails, you can pull the "Flight Recorder" logs for any entity and see exactly which system issued which command at which tick.

## 3. The Anatomy of a Tick
Every tick (e.g., `0.1` seconds) follows a strict sequence of phases to ensure determinism.

1. **Environment Phase:** Updates local conditions for all entities.
   *   *Systems:* `EnvironmentSystem` (Queries `TerrainOracle` for height, calculates International Standard Atmosphere air density based on altitude).
2. **Perception Phase:** Entities look at the world and update their internal tracks.
   *   *Systems:* `SensorSystem` (Calculates Radar/Sonar SNR, checks Line-of-Sight, adds/removes targets to `DetectionComponent`).
3. **Lifecycle Phase:** Manages internal entity states and timers.
   *   *Systems:* `WeaponStageSystem` (Burns boosters, drops stages), `HealthSystem` (Processes damage).
4. **Decision Phase:** The "Brains". Entities decide what they want to do based on their perceptions.
   *   *Systems:* `CombatSystem` (Pulls the trigger if a target is in range), `GuidanceSystem` (Calculates Pro-Nav interception paths for missiles).
5. **Force/Physics Phase:** The physical reality.
   *   *Systems:* `AeroSystem` (Calculates Lift/Drag based on air density and velocity), `PhysicsSystem` (Integrates Thrust + Drag + Lift + Gravity via F=ma to determine new velocities and positions).
6. **Final Sync:** 
   *   The `SpatialGrid` is updated with the new entity positions to accelerate spatial queries (e.g., `getNearbyEntities`) in the next tick.

## 4. How to Add a New Feature

If you need to add a new physical or tactical concept to the engine, follow this pattern:

### Step 1: Define the Data (Component)
Create a new Component in `src/engine/components/`. Keep it pure data.
```typescript
// src/engine/components/Fuel.ts
export class FuelComponent implements IComponent {
    readonly type = 'FuelComponent';
    constructor(public currentKg: number, public burnRateKgPerSec: number) {}
}
```

### Step 2: Define the Intent (Command)
Create a new Command in `src/engine/core/Command.ts`. Give it a logical priority.
```typescript
export class ConsumeFuelCommand extends Command {
    constructor(entityId: EntityId, public amount: number) {
        super(entityId, 20); // Priority 20
    }
}
```

### Step 3: Write the Logic (System)
Create a new System in `src/engine/systems/`. It reads the components and returns the commands.
```typescript
// src/engine/systems/FuelSystem.ts
export class FuelSystem implements ISystem {
    readonly name = 'FuelSystem';
    public async process(world: IWorldView, dt: number): Promise<Command[]> {
        const commands = [];
        for (const entity of world.getEntities()) {
            const fuel = entity.getComponent(FuelComponent);
            if (fuel && fuel.currentKg > 0) {
                commands.push(new ConsumeFuelCommand(entity.id, fuel.burnRateKgPerSec * dt));
            }
        }
        return commands;
    }
}
```

### Step 4: Apply the Command (Resolver)
Open `src/engine/core/World.ts` and add a block in `resolveCommands` to handle the actual state mutation.
```typescript
// Inside World.ts -> resolveCommands()
if (cmd instanceof ConsumeFuelCommand && fuel) {
    fuel.currentKg = Math.max(0, fuel.currentKg - cmd.amount);
    if (fuel.currentKg === 0) {
        // e.g. emit EngineFlameout event
    }
}
```

### Step 5: Register and Test
Add your system to `MatchService` (or test runner) and write an isolated unit test proving the command triggers under the right conditions.

## 5. Performance Considerations
While V3 prioritizes architecture over raw buffer manipulation, performance is maintained via:
1. **Spatial Partitioning:** The 10km `SpatialGrid` prevents O(N^2) loops in Sensor and Collision checks.
2. **Asynchronous Paging:** The `EnvironmentSystem` can asynchronously fetch terrain tiles (`.wgt` files) before the physics step, preventing disk I/O from stalling the main thread.
3. **Flat Prototypes:** Components are implemented as standard classes.    V8 (Node.js engine) is highly optimized for iterating over arrays of consistent objects.

# War-Games Engine V3: Expansion to C:MO Professional Standard

Building a simulation engine capable of rivaling **Command: Modern Operations (C:MO) Professional Edition** requires transitioning from a pure physics-and-sensor sandbox into a **massive data-driven, multi-domain C4ISR and doctrinal simulator**. 

C:MO’s strength is not just in calculating whether a missile hits; it is in modeling the entire kill chain: from national-level satellite reconnaissance down to the specific radar band of a ship's illuminator, governed by strict Rules of Engagement (ROE) and logistics.

## 7.1 The Data Foundation (The "DB3000" Equivalent)
C:MO is fundamentally a database engine attached to a physics simulator. You cannot hardcode platforms. You need a massive, highly structured, open-source intelligence (OSINT) database.

*   **Platform Schemas:** Every aircraft, ship, submarine, and facility must be defined by a strict JSON/SQL schema.
*   **Mounts & Magazines:** Platforms don't just "have weapons." They have specific mounts (e.g., Mk41 VLS, Mk45 5-inch gun) with arc limits (e.g., cannot fire aft), fed by specific magazines with limited capacities and reload rates.
*   **Signatures Matrix:** Entities need multi-spectrum signatures.
    *   *Radar Cross Section (RCS):* Front, Side, Rear, Top, Bottom values.
    *   *Infrared (IR):* Forward (skin friction) vs. Rear (exhaust plume).
    *   *Acoustic:* Active/Passive sonar signatures, factoring in cavitation speeds.
    *   *Visual:* Optical cross-section.

## 7.2 Advanced Environment & Oceanography
To simulate naval and anti-submarine warfare (ASW) accurately, the environment must be an active participant in the simulation, not just a static backdrop.

*   **Bathymetry & Sound Speed Profiles (SSP):** 
    *   *Feature:* Model the ocean in 3D layers. Water temperature, salinity, and pressure change with depth, bending sound waves.
    *   *ECS Implementation:* `OceanographySystem` that calculates Thermoclines, Surface Ducts, and Convergence Zones (CZ). A submarine below the thermal layer should be invisible to a surface ship's hull sonar.
*   **Dynamic Weather & Atmospherics:**
    *   *Feature:* Cloud cover, rain, and sea state.
    *   *Impact:* Heavy rain degrades X-band radar and IR sensors. High Sea State (e.g., Sea State 5) creates radar clutter (hiding small boats) and slows down surface ships.
*   **Orbital Mechanics (Space Domain):**
    *   *Feature:* Satellites cannot be steered like aircraft. They follow Keplerian orbits.
    *   *ECS Implementation:* `OrbitalPhysicsSystem` using Two-Line Element (TLE) sets to predict satellite passes for IMINT (Image Intelligence) and ELINT (Electronic Intelligence).

## 7.3 The Electromagnetic (EM) Spectrum & Sensors
Moving beyond basic Signal-to-Noise Ratio (SNR), the engine must model the nuances of the EM spectrum.

*   **Spectral Bands & Frequency:** Radars must operate on specific bands (L, S, C, X, Ku). A stealth fighter might be invisible to an X-band targeting radar but visible to an L-band early warning radar.
*   **ESM & ELINT (Electronic Support Measures):**
    *   *Feature:* Passive sensors that listen for enemy radar emissions.
    *   *Mechanic:* If a ship turns on its search radar, it gives away its position to enemy ESM long before the radar can detect the enemy (the "Flashlight in a Dark Room" effect).
*   **Advanced Radar Mechanics:**
    *   *Doppler Notching:* Aircraft flying perfectly perpendicular to a Pulse-Doppler radar (beaming) blend into the ground clutter and drop off the scope.
    *   *Illumination:* Semi-Active Radar Homing (SARH) missiles require the firing platform to keep a continuous "Continuous Wave" (CW) radar lock on the target until impact.
*   **Acoustics (Sonar):**
    *   *Types:* Hull-mounted, Towed Arrays (blind in the "baffles" behind the ship), Dipping Sonars (helicopters), Sonobuoys (LOFAR/DICASS).

## 7.4 Doctrine, AI, and C4ISR (The Brains)
Professional sims require entities to act autonomously based on strict military doctrine.

*   **Rules of Engagement (ROE):**
    *   *States:* Weapons Free, Weapons Tight, Weapons Hold.
    *   *Identification:* Targets must be classified (Unknown, Assumed Friend, Hostile) based on ESM emissions, flight profiles, or visual ID before firing.
*   **WRA (Weapon Release Authorization):**
    *   *Feature:* Granular logic for how AI uses weapons.
    *   *Example:* "Against a 4th-Gen Fighter, fire 2x AIM-120 missiles at 75% of maximum aerodynamic range. Against an MPA (Maritime Patrol Aircraft), fire 1x missile at max range."
*   **EMCON (Emissions Control):**
    *   *Feature:* Strict rules on when platforms are allowed to emit active radar/sonar.
    *   *Example:* A Carrier Strike Group operates in EMCON A (total silence), relying on airborne AWACS for the radar picture.
*   **The Common Operating Picture (COP) & Datalinks:**
    *   *Feature:* Entities do not act on their own `DetectionComponent`. They share tracks via Link 16 or CEC (Cooperative Engagement Capability).
    *   *Mechanic:* If the communications node (e.g., a satellite or relay aircraft) is destroyed or jammed, the network fragments, and units lose situational awareness.

## 7.5 Logistics, Facilities, and Turnaround
Wars are won on logistics. A professional sim models the downtime.

*   **Airbase Operations:**
    *   Aircraft require specific runway lengths.
    *   *Turnaround Times:* Landing, taxiing, refueling, rearming, and pre-flight checks take hours, not seconds.
    *   *Loadouts:* Aircraft must be assigned specific weapon loadouts (e.g., "Strike - Heavy", "CAP - Long Range") which alter their weight, drag (AeroComponent), and RCS.
*   **Underway Replenishment (UNREP):** Ships must rendezvous with supply vessels to restock VLS tubes and aviation fuel.

## 7.6 Component-Level Damage & Degradation
HP bars are insufficient for a professional sim.

*   **System-Level Damage:** A missile impact doesn't just reduce HP; it destroys specific components. 
    *   *Example:* A near-miss fragmentation warhead destroys a ship's SPY-1 radar arrays. The ship is still floating and has engine power, but is entirely blind and cannot guide its SAMs.
*   **Mobility Kills vs. Firepower Kills:** Modeling flooding, fires, and structural degradation over time.

## 7.7 "Professional" Analytical Features
To be a "Pro" engine used by defense contractors or analysts, the engine must interface with the outside world.

*   **Headless Monte Carlo Execution:** The ability to run the engine without a UI, executing a scenario 10,000 times at 100x speed with slight variations (e.g., varying missile seeker reliability by +/- 5%) to generate statistical kill probabilities.
*   **DIS / HLA Integration:** Distributed Interactive Simulation (IEEE 1278) and High Level Architecture (IEEE 1516). The engine must be able to send and receive UDP packets to sync its entities with *other* military simulators on a secure network.
*   **Scripting Engine:** A Lua (or TypeScript) API that allows scenario designers to write custom triggers (e.g., `If Unit X enters Zone Y, change EMCON to Active and spawn 4x MiG-29s`).
*   **Telemetry Export:** Every `Command` and state change recorded by your `Tracer` must be exportable to CSV, JSON, or a time-series database (like InfluxDB) for post-mission analysis in tools like Tacview.


# Navigation & Formation System.

Here is the revised strategy that incorporates Formations and Station Keeping:

## Updated Strategy: Navigation & Formation System

1. Define the Data Components
    * NavigationComponent: Stores the absolute Waypoint list, activeWaypointIndex, and a navState (e.g., 'Waypoint', 'Formation', 'Loiter').
    * FormationComponent: Added to entities that are followers. It holds:
        * leaderId: The Entity ID of the unit they are following.
        * stationOffset: A 3D vector representing their assigned position relative to the leader (e.g., "2 miles off the starboard bow, +1000
            ft altitude").
        * formationType: e.g., 'Line Abreast', 'Wedge', 'Custom'.
        * sprintDrift: Boolean. If true, the unit sprints to catch up and drifts to fall back (common in naval ASW screens). If false, it
            tries to exactly match the leader's speed.

2. Define the Intents (Commands)
    * AddWaypointCommand / ClearWaypointsCommand.
    * JoinFormationCommand (Assigns a leader and a station offset).
    * BreakFormationCommand (Removes the FormationComponent or sets navState back to individual waypoints).

3. Implement the Logic (FormationSystem & WaypointSystem)
    * We will implement these as two distinct systems running in the Decision Phase to maintain single-responsibility:
        * WaypointSystem: Only processes units where navState === 'Waypoint'. It calculates heading/speed to the next absolute geographic
            coordinate.
        * FormationSystem: Processes units with a FormationComponent. 
            * It queries the TransformComponent and KinematicsComponent of the leader.
            * It calculates the absolute world coordinate of the assigned station based on the leader's current position, heading, and the
                follower's stationOffset.
            * It acts as a dynamic waypoint generator: It issues a SetHeadingCommand to steer towards the station, and calculates a dynamic
                SetThrustCommand (accelerate if behind station, decelerate if ahead).

4. The Interaction Loop
    * When the UI assigns a Carrier (the Leader) to follow Waypoints, the WaypointSystem steers the Carrier.
    * The Escorts (the Followers) have their FormationSystem continually read the Carrier's new position and heading, dynamically adjusting
        their own headings and speeds to maintain their relative stations around the Carrier.

This approach creates a robust hierarchy. The user only needs to plot waypoints for the "Lead" unit of a Task Force, and the ECS seamlessly
handles the station-keeping for the rest of the group.

# Data-Extractor Subsystem Specification

## 1. Architectural Overview
The `data-extractor` is an offline, Node.js/TypeScript CLI tool. It does not run during the simulation. Its sole purpose is to ingest unstructured OSINT text files, use a local LLM (e.g., Qwen-2.5-14B via Ollama) to extract data into strict Zod schemas, validate the physics, and output compiled JSON profiles for the `ProfileRegistry`.

### Pipeline Stages:
1.  **Ingestion:** Read raw `.txt` or `.md` files.
2.  **LLM Extraction:** Force the LLM to call a `SavePlatformProfile` tool, ensuring the output strictly matches our Zod schemas.
3.  **Physics Sanity Validation:** Run programmatic checks (e.g., "Does thrust exceed weight?", "Is RCS physically possible?") to catch LLM hallucinations.
4.  **Variant Consolidation:** Group similar platforms (e.g., F-16A, F-16C) and extract common traits into a Base Profile to save memory.
5.  **Export:** Write to `data/db3000/profiles.json`.


### Summary of the Data-Extractor Workflow
1.  You drop 1,000 text files into `data/raw_osint/`.
2.  You run `npx ts-node data-extractor/main.ts`.
3.  The script feeds each file to Qwen-14B.
4.  Qwen-14B uses the `SaveProfileTool` to output strict Zod-validated JSON.
5.  The `SanityValidator` checks if the LLM hallucinated a Mach 5 cargo ship.
6.  The `Consolidator` groups all the F-16 files together into a single Base Platform with multiple Variants.
7.  The final output is saved to `db3000_compiled.json`, ready to be loaded by your `ProfileRegistry` in Engine V3.




Transforming the War-Games Engine V3 from a high-fidelity physics sandbox into a Professional Standard simulator (comparable to C:MO) requires a fundamental shift in scope. The focus moves from purely calculating *how* a missile flies to determining *if* the Strike Group Commander legally authorizes the launch based on degraded, fused sensor data.

Here is a breakdown of how the V3 Command-Driven ECS architecture expands to meet this standard.

### 1. The Data Foundation: The "DB3000" Approach
A professional military simulation cannot rely on hardcoded platform logic. It must be built on a massive, highly structured database—a comprehensive open-source compilation of information on modern military equipment. 

In V3, your `ProfileRegistry` and `EntityManager` become the bridge to this data:
* **Decoupled Loadouts:** An aircraft entity doesn't inherently possess weapons. The database schema defines specific *Mounts* (e.g., Mk41 VLS) and *Magazines* with arc limits and reload rates. 
* **Signature Matrices:** Instead of a generic visibility flag, entities require multi-spectrum signatures. The database must support calculating theoretical detection ranges based on specific radar parameters and the target's aspect-dependent Radar Cross Section (RCS). 



### 2. C4ISR and Electromagnetic Battle Management
The engine must simulate the unseen war. Electromagnetic battle management in a congested and contested environment becomes just as critical as the physical flight models.

* **Spectral Specificity:** Radars in V3 must be assigned operating bands (L, S, C, X, Ku). A stealth platform optimized against X-band targeting radars might still be illuminated brightly by an older L-band early warning radar.
* **Electronic Support Measures (ESM):** The "flashlight in a dark room" mechanic. If a ship activates its search radar, it emits a signal that enemy passive ESM sensors can detect from far beyond the radar's actual tracking range.
* **The Common Operating Picture (COP):** Entities should not act on their own localized `DetectionComponent`. Instead, a `CommSystem` must fuse individual detections via simulated datalinks (like Link 16) into a shared network. If the communications node is jammed or destroyed, the network fragments and units lose situational awareness.



### 3. Doctrine, WRA, and the Decision Phase
The V3 tick architecture explicitly isolates the `Decision Phase`. To achieve professional realism, the `CombatSystem` and `GuidanceSystem` must be restricted by doctrinal logic rather than simply pulling the trigger when a target is in range.

* **Rules of Engagement (ROE):** Targets must pass through an identification matrix (Unknown, Assumed Friend, Hostile). An entity under "Weapons Tight" ROE will refuse to generate a `FireWeaponCommand` against an unclassified track, even if it is physically in range.
* **Weapon Release Authorization (WRA):** Granular AI logic dictates resource management. The doctrine might dictate firing exactly two AIM-120 missiles at a 4th-generation fighter at 75% of maximum range, but only one against a slower maritime patrol aircraft.
* **Emissions Control (EMCON):** A Carrier Strike Group operating under EMCON Alpha (total silence) forces the system to suppress all active radar and sonar commands, relying entirely on passive ESM and datalinks.



### 4. Advanced Environment and Oceanography
The environment must actively interact with the `SensorSystem` and `PhysicsSystem`. 

* **3D Bathymetry & Acoustics:** The ocean is modeled in layers. Water temperature, pressure, and salinity create thermoclines. If a submarine entity dives beneath the thermal layer, the sound waves bend, rendering it invisible to a surface ship's hull-mounted sonar. 
* **Atmospherics & Sea State:** High winds generate heavy sea states, creating radar clutter that masks small surface vessels. Heavy rain physically attenuates X-band and Ku-band transmission ranges.

### 5. Component-Level Degradation
A simple health bar is insufficient. The V3 `HealthSystem` must process targeted damage. If a near-miss fragmentation warhead destroys an Arleigh Burke's SPY-1 radar arrays, the ship retains its buoyancy and engine power but is completely blind. It suffers a "mission kill" and loses the ability to guide its surface-to-air missiles, even with 80% of its overall structural HP remaining.
